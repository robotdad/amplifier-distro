# Amplifier Directory Contract

> Canonical specification for the `~/.amplifier/` filesystem layout.
> All interfaces, tools, and modules MUST conform to this contract.

## Overview

Amplifier stores all persistent state under a single root directory,
defaulting to `~/.amplifier/`. This can be overridden via the
`AMPLIFIER_HOME` environment variable.

## Directory Layout

```
~/.amplifier/                        # AMPLIFIER_HOME
  settings.yaml                      # User configuration (global)
  cache/                             # Module and bundle cache
    <name>-<hash>/                   # Cached git clone (shallow)
    install-state.json               # Module installation tracking
  projects/                          # Per-project state
    <project-slug>/                  # Derived from working directory name
      sessions/                      # Session storage
        <session-id>/                # One directory per session
          transcript.jsonl           # Conversation transcript
          events.jsonl               # System events log
          metadata.json              # Session metadata
          handoff.md                 # Session handoff summary (auto-generated)
      recipe-sessions/               # Recipe execution state
  memory/                            # Shared memory store
    memory-store.yaml                # Facts, preferences, learnings
    work-log.yaml                    # Active work, pending decisions
  modes/                             # User-defined modes
    <mode-name>.md                   # Mode definition files
  skills/                            # User-defined skills
    <skill-name>/                    # Skill packages
  bundles/                           # Registered bundle references
  AGENTS.md                          # User-global agent instructions
```

## Path Derivation

### Session ID

Session IDs are generated by the kernel and are globally unique. Format
is implementation-defined but must be filesystem-safe.

### Cache Entry Names

Cache entries use `<name>-<hash>` format where:
- `name` is derived from the git repository name
- `hash` is a content-addressable hash for deduplication

## Ownership and Lifecycle

| Directory | Owner | Created By | Cleaned By |
|-----------|-------|------------|------------|
| `settings.yaml` | User / `amplifier init` | `amplifier init` or manual | User |
| `cache/` | Cache subsystem | Module/bundle loader | `amplifier reset --remove cache` |
| `projects/` | Session subsystem | Session creation | User or `amplifier reset` |
| `memory/` | Memory subsystem | First memory write | User |
| `modes/` | User | Manual | User |
| `skills/` | User / skill loader | Manual or skill install | User |
| `bundles/` | Bundle subsystem | `amplifier bundle add` | `amplifier bundle remove` |
| `AGENTS.md` | User / AI | Manual or AI-generated | User |

## Concurrency

- Session directories are owned by a single session process. No locking needed.
- `settings.yaml` is read at startup. Concurrent writes are not supported.
- `cache/` entries use atomic replacement (write to temp, rename) to avoid partial states.
- `memory/` files may be read by multiple sessions. Write serialization is the caller's responsibility.
- `handoff.md` is written at session end inside that session's directory. No concurrent access expected.

## Security

- `settings.yaml` should be readable only by the owner (chmod 600 recommended).
- API keys should be stored in environment variables, NOT in `settings.yaml`.
- Session transcripts may contain sensitive information. The `projects/` directory should have restricted permissions.
- The `cache/` directory contains cloned git repos. Access permissions follow the parent directory.

## Environment Variables

| Variable | Purpose | Default |
|----------|---------|---------|
| `AMPLIFIER_HOME` | Override the root directory | `~/.amplifier` |
| `AMPLIFIER_CACHE_DIR` | Override cache location | `$AMPLIFIER_HOME/cache` |
| `AMPLIFIER_PROJECTS_DIR` | Override projects location | `$AMPLIFIER_HOME/projects` |

## Settings File (`settings.yaml`)

The file `$AMPLIFIER_HOME/settings.yaml` is the user-level configuration
file. It is read at session startup. Applications MAY implement additional
scoping (e.g. per-project or per-session overrides), but those are
app-layer concerns and not part of this contract.

### Schema

```yaml
# ── Bundle Configuration ──────────────────────────────────
bundle:
  active: <name>                  # Currently active bundle name
  added:                          # User-added bundles (name → source URI)
    <name>: <source-uri>          # e.g. my-bundle: git+https://github.com/org/repo

# ── Provider Configuration ────────────────────────────────
config:
  providers:                      # List of provider configurations
    - module: <module-id>         # Required. e.g. "provider-anthropic"
      source: <source-uri>        # Optional. Git URL or file:// path
      config:                     # Provider-specific settings
        api_key: "${ANTHROPIC_API_KEY}"     # Env var expansion (see below)
        default_model: "claude-sonnet-4-20250514"
        base_url: "https://..."

# ── Source Overrides ──────────────────────────────────────
sources:
  modules:                        # Module source overrides
    <module-id>: <source-uri>     # e.g. tool-bash: file:///home/user/repos/tool-bash
  bundles:                        # Bundle source overrides
    <bundle-id>: <source-uri>
```

### Key Sections

#### `bundle`

Manages which bundle is active and which user-added bundles are available.

- **`bundle.active`** — Name of the currently active bundle. Set by
  `amplifier bundle use <name>`, read at session startup to determine
  which bundle to load.
- **`bundle.added`** — Dict of `name → source-uri` for bundles the user
  has registered via `amplifier bundle add`. Source URIs follow the
  standard URI format (git+https, file://, etc.).
Applications MAY store additional keys under `bundle` (e.g. app-specific
bundle lists). Unknown keys are ignored.

#### `config.providers`

A list of provider configuration dicts. Each entry must have a `module`
field used as the match key. The merge behavior depends on whether the
active bundle already declares providers:

- **Bundle has providers** — Settings can only *update* providers the
  bundle already declared (matched by `module`). New providers from
  settings are silently ignored. The `config` sub-dict is deep-merged
  so that bundle-specific keys (e.g. `debug`, `default_model`) survive
  when settings only provide runtime keys (e.g. `api_key`, `base_url`).
  All other top-level fields (`source`, etc.) are replaced by settings
  values when present.

- **Bundle has no providers** (provider-agnostic) — The full provider
  list from settings is used as-is.

#### `sources`

Module and bundle source overrides. These let you point Amplifier at a
local checkout or alternate git URL instead of the default source
declared in the bundle.

Source override precedence (first match wins):
1. `config.providers[].source` — per-provider source (most specific)
2. `sources.modules.<id>` / `sources.bundles.<id>` — general overrides

### Environment Variable Expansion

String values in `config.providers[].config` support `${VAR}` and
`${VAR:default}` syntax:

```yaml
config:
  providers:
    - module: provider-anthropic
      config:
        api_key: "${ANTHROPIC_API_KEY}"            # Required, empty if unset
        base_url: "${ANTHROPIC_BASE_URL:}"         # Optional, pruned if empty
        default_model: "${MODEL:claude-sonnet-4-20250514}"  # Falls back to default
```

After expansion, dict entries whose values resolve to empty string `""`
are **pruned**. This prevents empty env vars (e.g. `ANTHROPIC_BASE_URL=''`)
from overriding provider defaults with blank values.

Expansion is applied **after** the settings-bundle merge, so bundle-owned
config keys survive when a `${VAR}` resolves to empty.

### Error Handling

Settings loading is fully defensive — no exceptions propagate:

| Condition | Behavior |
|-----------|----------|
| File missing | Silently skipped (empty dict) |
| Invalid YAML | Logged as warning, silently skipped |
| `config.providers` not a list | Returns empty provider list |
| Unknown top-level keys | Ignored (forward-compatible) |

### Security

- **API keys** — Store in environment variables, reference via `${VAR}`
  syntax. Never store raw secrets in `settings.yaml`.
- **File permissions** — `chmod 600` recommended for global settings.

## Compatibility

This contract is versioned with amplifier-foundation. Changes to the directory
layout MUST be backward-compatible or accompanied by a migration path.

Interfaces that conform to this contract can interoperate seamlessly —
sessions started in the CLI can be resumed in the TUI, handoffs written
by hooks can be read by any interface.
